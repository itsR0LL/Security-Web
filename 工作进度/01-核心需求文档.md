# Security Studio 核心需求文档 v1.0

更新时间：2026-06-02

## 1. 项目定位

Security Studio 是面向个人博客站点的私有安全态势与攻击行为分析平台。

它的核心不是公开大屏，也不是企业级 SIEM / EDR，而是服务于 `r0l1dehome.asia` 的个人安全监控、攻击可视化、规则研判与后续 AI 辅助分析。

当前项目优先作为独立项目在 `E:\Project\Secrity Studio` 中运行，稳定后再考虑接入博客管理后台 `my-blog-manager`。

## 3. 核心使用者

当前只面向博客管理员本人。

一期不做复杂多角色权限系统。设置、同步、事件查看、告警文本生成都默认由管理员使用。

## 4. 核心问题

Security Studio 要解决的问题分为三层：

### 监控层

把 Cloudflare、博客访问、安全事件、后续服务器日志接入同一套可视化和分析平台。

### 研判层

把单条日志转成可理解的安全事件，并通过规则判断攻击类型、风险等级、来源特征、处理结果。

### 推理层

基于一段时间内的攻击行为聚合，识别攻击工具、攻击习惯、异常来源、规则盲区，并让 AI 辅助生成规则更新建议。

## 5. 下一阶段核心方向

下一阶段不再继续堆叠 HUD 或单纯做视觉效果，核心方向改为：

`Rule JSON + Event Feature + Attack Cluster`

目标链路：

```text
日志事件
-> 标准化事件
-> 规则命中
-> 特征提取
-> 攻击聚合
-> AI 辅助归类与建议
-> 人工审核
-> 规则版本更新
-> 可视化反馈
```

## 6. AI 的边界

AI 在当前阶段只做三件事：

- 对攻击日志进行解释。
- 对相似攻击行为进行聚合归类。
- 根据已有规则格式给出规则更新建议。

AI 当前阶段不做：

- 不自动启用规则。
- 不自动封禁。
- 不直接改 Cloudflare 防护配置。
- 不把推测当作真实攻击者身份。

所有 AI 输出都必须经过人工审核、回放验证和版本记录。

## 7. 攻击者追踪的定义

项目后期可以做攻击来源与攻击链分析，但必须采用安全、可验证的定义。

这里的“找到攻击者”不表示确认真实个人身份，而是尽可能还原：

- 使用了哪些代理、节点、ASN、国家/地区。
- 是否存在僵尸网络、扫描器、自动化工具特征。
- 是否使用相似 UA、路径序列、请求节奏、参数组合。
- 攻击行为是否来自同一基础设施或同一攻击流程。
- 当前规则是否存在被绕过的痕迹。

最终输出应为“攻击基础设施画像”和“攻击行为画像”，并带有证据与置信度。

## 8. 一期之后的 MVP 目标

### 必须推进

- 建立可版本化的规则 JSON 结构。
- 给事件增加攻击特征字段。
- 建立攻击聚合逻辑。
- 新增攻击分析页面。
- 在 `/security/situation` 中增加安全态势侧栏指标。
- 做 AI 规则建议的模拟链路。

### 暂缓推进

- AI 自动更新并启用规则。
- 主机操作系统日志接入。
- 完整攻击链还原。
- 攻击者真实身份归因。
- 自动阻断和自动处置。
- 大规模企业安全平台能力。

## 9. 数据源约束

当前 Cloudflare 免费版能力有限，必须在产品表现中明确区分真实数据、样例数据、降级数据和旧数据。

需要遵守：

- 未配置 Token 时自动启用样例数据。
- Token 无效时显示 Token 无效。
- 权限不足时显示权限不足。
- 同步失败时显示同步失败。
- 旧数据可用时显示旧数据状态。
- 不把错误状态伪装成样例模式。

## 10. 成功标准

下一阶段完成后，Security Studio 应该能做到：

- 管理员能看到当前博客安全状态。
- 管理员能查看事件证据和详情。
- 系统能根据规则给事件打上风险与攻击类型。
- 系统能按来源、工具、行为节奏进行聚合。
- 页面能解释“最近哪些攻击更集中、哪些规则被频繁命中、哪些攻击特征值得新增规则”。
- AI 的建议可以被人工审核、测试、记录、回滚。
