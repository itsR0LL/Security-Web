# 博客安全态势监控平台核心需求文档 v0.6

## 1. 项目定位

本项目定位为一个私有的博客安全态势监控平台，面向个人博客资产提供访问监控、安全事件分析、风险态势可视化和后续 AI 分析预留能力。

它不是公开展示页，也不是企业级 SIEM。一期目标是作为可独立运行的安全后台工具，后续可以嵌入或融合到现有博客管理后台中，成为后台中的安全态势页面。

## 2. 现有系统背景

已查看本地项目：

- 公开博客站：`E:\Project\personalBlogweb`
- 私有管理后台：`E:\Project\my-blog-manager`

两个项目主要技术栈一致：

- Next.js 16
- React 19
- TypeScript
- Tailwind CSS 4
- lucide-react
- framer-motion
- npm scripts: `dev`、`build`、`start`、`lint`

管理后台额外包含：

- Python FastAPI 后端：`cms_core`
- 本地启动器：`launcher.py`
- 本地 API 鉴权：`X-Manager-Token`
- 本地运行边界：管理后台不部署到公网，仅作为 sibling app 本地运行

因此，安全平台一期建议沿用现有模式：

- 前端：Next.js + React + Tailwind
- 后端：本地 FastAPI 服务
- 存储：本地 SQLite 起步，必要时保留原始事件 JSON 字段
- 访问边界：默认私有、本地优先、不可公开暴露完整安全数据

## 3. 核心用户

一期只考虑一个主要用户：

- 博客管理员，也就是项目所有者本人

暂不设计复杂多角色体系，但保留基础访问保护：

- 本地管理 token
- 或后续接入博客管理后台已有本地鉴权方式

## 4. 核心目标

一期需要解决的问题：

1. 让管理员知道博客最近的访问态势。
2. 让管理员看到哪些请求可能是扫描、探测、攻击或异常访问。
3. 让管理员能够通过图表快速判断风险趋势、来源 IP、异常路径和攻击类型。
4. 让系统先具备规则分析能力，并为后续大模型分析预留接口和数据结构。
5. 让项目可以独立运行，同时后续能够作为博客管理后台的安全模块接入。

- 项目先在当前 `E:\Project\Secrity Studio` 目录内作为独立项目开发，稳定后再考虑接入 `E:\Project\my-blog-manager`。
- Cloudflare 配置入口一期放在安全平台自己的 `/security/settings`，后续再合并到博客管理后台设置页。
- 首页固定包含：访问态势、攻击态势、异常来源、最近事件、流量分析。
- 一期直接做 3D 仿真地球模块，但不阻塞列表、统计和核心分析主流程。
- 3D 仿真地球展示来源点，并用飞线连接到博客站点位置。
- 风险等级一期使用内置默认规则，设置页只允许调整“高风险提醒阈值”。
- 后续预留页面化添加/编辑规则的能力。
- 一期加入样例数据模式，未配置 Cloudflare Token 时也能预览页面、地球可视化和筛选功能。
- 3D 地球目的地固定为成都。
- 流量分析优先级为：请求量、带宽、缓存命中、状态码、热门路径、来源国家/地区。
- 样例数据模式只在未配置 Cloudflare Token 时自动启用。
- 一期加入数据同步状态面板，展示最近同步时间、同步结果、API 错误和本地数据量。
- 首页采用“一屏总览 + 下钻列表”信息架构，不做纯沉浸式 3D 首页。
- 允许使用 Three.js 相关依赖实现 3D 地球。
- 数据同步失败时保留上一次成功数据，并展示“数据可能过期”提示。
- 配置 Cloudflare Token 后需要自动进行权限检测，检查 Zone、Analytics、Security Events 是否可读。
- 事件列表默认按时间倒序排序，同时必须提供风险等级、事件类型、来源、时间范围等筛选能力。
- 核心需求文档之后继续产出《技术方案文档》和《页面原型说明》。

### 5.1 安全态势首页

首页展示博客安全与访问概览：

- 最近 6 小时、24 小时、7 天访问量
- 异常请求数量
- 高风险事件数量
- Cloudflare 安全事件数量
- Top 来源 IP
- Top 访问路径
- Top User-Agent
- 状态码分布
- 风险等级分布
- 攻击或扫描类型分布
- 最近高风险事件时间线
- 流量分析：请求量、缓存命中/未命中、源站流量、边缘侧流量、带宽趋势、状态码趋势、热门路径趋势
- 3D 仿真地球可视化：按访问、攻击、安全事件类型、风险等级等筛选维度，展示来源点，并用飞线连接到成都

刷新方式：

- 支持手动刷新
- 支持每 6 小时被动刷新一次
- 暂不要求秒级实时推送
- 同步失败时保留上一次成功数据，并在首页与同步状态面板显示数据可能过期。

### 5.2 数据接入

一期优先接入 Cloudflare 数据，原因：

- Cloudflare 能看到边缘侧 HTTP 请求、安全事件、WAF/规则命中等信息。
- 被 Cloudflare 拦截、挑战或标记的请求可能不会进入博客源站日志。
- 官方 GraphQL Analytics API 可查询 HTTP 请求、Firewall/Security Events 等数据。

一期数据源优先级：

1. Cloudflare GraphQL Analytics API：用于拉取聚合访问趋势、安全事件和最近事件。
2. Cloudflare Security Events：用于查看 Cloudflare 安全产品已处理或标记的请求。
3. 博客源站或应用日志：作为补充，用于查看进入应用后的请求和错误。
4. 手动导入样例日志：用于无 Cloudflare API 或开发测试时验证功能。

一期接入策略：

- 优先做 Cloudflare 只读接入。
- Cloudflare API Token 仅授予读取 analytics/security events 所需权限。
- Token、Zone ID 等敏感配置只保存在本地后端配置或本地 SQLite，不暴露到前端源码和公开博客仓库。
- 博客源站日志接入作为补充能力，不作为一期首个阻塞项。
- 由于 Cloudflare 免费版 Security Events 的历史窗口较短，一期需要按 6 小时刷新任务把 Cloudflare 数据沉淀到本地 SQLite，避免只依赖 Cloudflare 控制台历史查询。

一期本地保留策略：

- 原始事件：保留 90 天。
- 聚合统计：长期保留。
- 90 天前的原始事件可按天汇总为聚合数据后清理。
- 聚合维度至少包括：日期/小时、事件类型、风险等级、国家/地区、来源 IP、访问路径、Cloudflare action。

Cloudflare 免费版相关约束：

- Security Events 在免费版可用，但历史数据窗口最多 24 小时，最大查询窗口也是 24 小时。
- Security Analytics 在免费版可用，历史数据窗口最多 7 天，最大查询窗口为 24 小时。
- Security Events 可能使用采样数据，事件列表不应承诺完整覆盖每一条请求。
- 普通访问统计更适合用聚合查询，不能把它当作完整原始访问日志。
- 某些字段是否可用需要通过 Cloudflare GraphQL `settings` 节点或实际查询结果确认，并在 UI 中做字段缺失降级。

对产品能力的影响：

- 不影响首页态势、趋势图、Top IP、Top 路径、状态码、风险分布等聚合展示。
- 不影响 3D 地球的基础展示，至少可以用国家/地区、来源 IP 或本地 IP 地理库生成来源点。
- 可能影响城市级精确定位、完整原始请求列表、长时间跨度的安全事件回溯、部分规则命中细节。
- 系统必须每 6 小时同步并保存本地 SQLite，否则免费版 24 小时后的 Security Events 明细可能无法再查询。

地理定位与飞线降级策略：

- 优先使用 Cloudflare 返回的城市、经纬度、地区、国家等字段。
- 如果 Cloudflare 返回来源 IP，但缺少城市或经纬度，则使用本地 IP 地理库补全。
- 如果只能拿到省/州/地区，则在该省/州/地区范围内生成稳定伪随机点。
- 如果只能拿到国家，则在国家范围内生成稳定伪随机点。
- 如果只有聚合国家统计，则展示国家级热力点或国家级飞线，不展示具体 IP 明细。
- 随机点必须基于事件 ID、IP 或国家/地区生成稳定结果，避免每次刷新位置跳动。
- 城市级飞线作为优先目标，但 UI 需要标记定位精度：城市、地区、国家、估算。

Cloudflare 官方文档参考：

- GraphQL Analytics API: https://developers.cloudflare.com/analytics/graphql-api/
- Cloudflare Logs: https://developers.cloudflare.com/logs/
- Security Events: https://developers.cloudflare.com/waf/analytics/security-events/
- Logpush HTTP destination: https://developers.cloudflare.com/logs/logpush/logpush-job/enable-destinations/http/

### 5.3 日志解析与事件归一化

系统需要把不同来源的数据统一成安全事件模型。

建议一期事件字段：

- `id`
- `timestamp`
- `source`
- `clientIp`
- `country`
- `asn`
- `host`
- `method`
- `path`
- `query`
- `statusCode`
- `userAgent`
- `referer`
- `rayId`
- `action`
- `ruleId`
- `ruleName`
- `eventType`
- `riskLevel`
- `confidence`
- `summary`
- `raw`

一期事件类型建议：

- 正常访问
- 爬虫访问
- 恶意扫描
- 目录探测
- 敏感路径探测
- 疑似注入
- 疑似 XSS
- 异常状态码
- 可疑 User-Agent
- Cloudflare WAF 事件
- Cloudflare Challenge 或 Block 事件

### 5.4 规则分析

大模型分析暂缓，但一期应先做规则分析，并预留 AI 分析接口。

规则分析至少支持：

- 敏感路径匹配，例如 `/admin`、`/.env`、`/wp-login.php`、`/phpmyadmin`
- 常见扫描器 User-Agent 匹配
- 高频 IP 访问识别
- 高频 404 识别
- 疑似 SQL 注入关键词识别
- 疑似 XSS 关键词识别
- Cloudflare action/rule 映射为风险等级

风险等级建议：

- `info`
- `low`
- `medium`
- `high`
- `critical`

规则配置策略：

- 一期使用内置默认规则。
- 设置页一期只允许调整高风险提醒阈值。
- 数据结构预留自定义规则表，后续支持在页面添加、启用、禁用和调整规则。
- 自定义规则二期再做，避免一期规则编辑器拖慢主流程。

### 5.5 事件列表与详情

事件列表支持：

- 按时间筛选
- 按风险等级筛选
- 按事件类型筛选
- 按来源 IP 筛选
- 按路径搜索
- 按 Cloudflare action 筛选

事件详情展示：

- 解析后的字段
- 原始日志或原始 Cloudflare 事件
- 规则命中原因
- 风险等级
- 后续 AI 分析占位区
- 管理员备注或处理状态可作为二期能力

### 5.6 高风险事件提示

一期不做自动告警发送。

一期设计为：

- 管理员可配置一个风险阈值，例如 `high` 及以上。
- 系统发现达到阈值的事件后，在页面中生成待发送消息。
- 管理员先复制或手动处理该消息。
- 外部发送渠道放到二期，后续可支持邮件、Server 酱、Telegram、企业微信、飞书、Bark 等。

## 6. 嵌入与运行方式

一期需要同时满足两个形态：

### 6.1 独立运行

安全平台可以作为单独项目启动：

- 独立 Next.js 前端
- 独立 FastAPI 后端
- 独立本地配置文件
- 独立本地数据存储

### 6.2 后台融合预留

后续可以作为博客管理后台中的安全模块：

- 作为 `/security`、`/security/events`、`/security/settings` 等后台路由
- 复用管理后台视觉风格
- 复用本地后端 token 模式
- 不并入公开博客根目录
- 不向公网公开完整安全事件数据

## 7. 非目标和二期能力

一期不做：

- 企业级 SIEM
- 多租户
- 复杂 RBAC 权限体系
- 自动封禁 IP
- 自动修改 Cloudflare WAF 规则
- 秒级实时推送
- 大规模日志集群
- 对外公开安全大屏
- 完整 AI 自动研判闭环
- 多站点资产管理

可作为二期：

- AI 分析与处置建议
- 自动告警渠道
- 手动封禁或 Cloudflare 规则联动
- 管理员处理状态与备注
- 安全周报/月报
- 多博客或多域名支持
- Logpush 到 HTTP 接收端
- 更完整的权限体系

## 9. 验收标准草案

一期可验收标准：

1. 管理员可以本地启动安全平台，并打开安全态势首页。
2. 管理员可以配置 Cloudflare API Token 和 Zone ID。
3. 管理员可以手动刷新最近安全数据。
4. 系统可以按 6 小时周期刷新数据。
5. 系统可以展示访问趋势、风险分布、Top IP、Top 路径和最近高风险事件。
6. 系统可以把 Cloudflare 数据或导入日志归一化为统一事件模型。
7. 系统可以基于规则识别至少 5 类常见风险事件。
8. 管理员可以进入事件列表并筛选高风险事件。
9. 管理员可以查看事件详情和原始数据。
10. 管理员可以在仿真地球上按访问、攻击、风险等级等维度筛选并查看事件分布。
11. 系统可以保留 90 天原始事件，并长期保留聚合统计。
12. 系统可以在未配置 Cloudflare Token 时启用样例数据模式。
13. 系统可以展示流量分析，包括请求趋势、状态码趋势、热门路径和缓存/源站相关指标。
14. 系统可以展示数据同步状态，包括最近同步时间、同步结果、API 错误和本地事件数量。
15. 3D 地球飞线目的地固定为成都；来源位置优先城市级，缺失时按地区/国家稳定降级。
16. 系统在配置 Cloudflare Token 后可以自动检测 Zone、Analytics、Security Events 读取权限。
17. 事件列表默认按时间倒序，并提供风险、类型、来源、时间、路径等筛选入口。
18. AI 分析区域可以展示占位状态，但不强制接入大模型。
