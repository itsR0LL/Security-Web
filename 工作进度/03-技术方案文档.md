## 3. 当前数据策略

数据来源：

- Cloudflare 免费版数据。
- 本地 SQLite。
- 样例数据 fallback。

数据保留：

- 原始事件保留 90 天。
- 聚合统计长期保留。

样例模式规则：

- 未配置 Cloudflare Token 时自动启用样例数据。
- Token 无效时显示 Token 无效。
- 权限不足时显示权限不足。
- 同步失败时显示同步失败。
- 旧数据可用时显示旧数据状态。

## 4. 下一阶段核心技术链路

下一阶段的技术主线是攻击研判与规则迭代。

```text
Cloudflare / 博客 / 后续服务器日志
-> EventNormalizer
-> FeatureExtractor
-> RuleMatcher
-> AttackAggregator
-> AiRuleAdvisor
-> ReviewWorkflow
-> RuleVersionStore
-> VisualizationFeedback
```

### EventNormalizer

把 Cloudflare 或其他来源日志转换为统一内部事件。

### FeatureExtractor

从事件中提取攻击特征。

### RuleMatcher

使用版本化 JSON 规则判断风险、分类和证据。

### AttackAggregator

把相似事件聚合成攻击行为组。

### AiRuleAdvisor

根据聚合结果生成规则更新建议。

### ReviewWorkflow

人工审核 AI 建议，决定是否进入规则版本。

### RuleVersionStore

保存规则版本、变更原因、测试结果、回滚信息。

### VisualizationFeedback

把攻击分类、规则命中、攻击习惯展示到 `/security/analysis` 和 `/security/situation`。

## 5. 规则 JSON 设计方向

规则必须可版本化、可测试、可回滚。

建议结构：

```json
{
  "id": "rule-example",
  "version": "1.0.0",
  "name": "示例规则",
  "enabled": true,
  "mode": "shadow",
  "scope": {
    "source": "cloudflare",
    "domain": "r0l1dehome.asia"
  },
  "conditions": {
    "path": [],
    "query": [],
    "userAgent": [],
    "method": [],
    "statusCode": [],
    "cloudflareAction": []
  },
  "classification": {
    "attackCategory": "scan",
    "attackSubtype": "path_probe",
    "severity": "medium"
  },
  "confidence": {
    "base": 0.6,
    "evidenceRequired": []
  },
  "evidence": [],
  "actions": {
    "alert": true,
    "block": false
  },
  "lifecycle": {
    "createdBy": "manual",
    "reviewStatus": "approved"
  }
}
```

规则字段上线前必须由代码中的类型定义固定，不允许前端和后端各自维护不同格式。

## 6. 事件特征字段方向

后续事件数据需要增加以下字段：

- `attack_category`
- `attack_subtype`
- `tool_signature`
- `behavior_fingerprint`
- `campaign_id`
- `rule_hits`
- `ai_cluster_id`
- `rule_version`
- `confidence`
- `location_precision`

这些字段用于事件详情、攻击聚合、地图筛选、AI 规则建议。

## 7. 攻击聚合维度

AttackAggregator 至少需要支持以下维度：

- 时间窗口。
- 来源国家、地区、城市。
- 来源 IP。
- ASN。
- 请求路径。
- 请求方法。
- 状态码。
- 查询参数模式。
- User-Agent 稳定性。
- 请求频率。
- 路径序列。
- Cloudflare action。
- 规则命中。
- 风险等级。
- 攻击类型。

输出不是单条事件，而是攻击行为组。

攻击行为组应包含：

- 事件数量。
- 时间范围。
- 主要来源。
- 主要路径。
- 主要工具指纹。
- 主要规则命中。
- 风险等级。
- 置信度。
- 代表性证据。

## 8. AI 辅助规则治理

AI 的输入应是聚合后的摘要，不是完整敏感日志。

AI 输入可以包含：

- 事件数量。
- 时间范围。
- 来源区域。
- 路径模式。
- UA 摘要。
- 状态码分布。
- Cloudflare action 分布。
- 已命中规则。
- 未命中但可疑的行为特征。

AI 输出必须是草案：

- 攻击类别建议。
- 规则 JSON 建议。
- 证据说明。
- 影响范围。
- 风险提醒。
- 需要人工确认的问题。

AI 输出不得直接写入启用规则。

## 9. 后续服务器日志链路

Cloudflare 日志只能看到边缘层信息。后续如果要分析绕过规则后的服务器行为，需要增加应用日志和主机日志。

建议顺序：

1. 博客应用层日志。
2. 管理后台访问日志。
3. Nginx 或反向代理日志。
4. 主机登录与进程行为日志。
5. 文件变更与命令执行记录。

这些内容需要独立权限、脱敏和存储策略，当前阶段不直接接入。

## 10. 开发顺序

### 第一轮

- 固定规则 JSON 类型。
- 扩展事件模型。
- 在样例数据中加入攻击分类与规则命中。
- 建立基础 RuleMatcher。

### 第二轮

- 建立 AttackAggregator。
- 增加聚合接口。
- 增加 `/security/analysis` 页面。
- 在 `/security/situation` 侧栏展示攻击习惯与规则趋势。

### 第三轮

- 接入 AI 规则建议草案。
- 建立人工审核状态。
- 建立规则版本记录。
- 增加 shadow mode 回放测试。

### 第四轮

- 接入更多日志源。
- 建立攻击流程分析。
- 输出基础设施画像与行为画像。
